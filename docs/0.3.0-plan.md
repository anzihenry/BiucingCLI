# BiucingCLI 0.3.0 Plan

## Current State

BiucingCLI `0.2.0` is already a capable scaffold generator, but most of the shipped depth lives inside the templates rather than in the CLI product surface.

Verified current functionality from `src/biucingcli/cli.py`, `src/biucingcli/templates.py`, `templates/*/template.json`, and `tests/test_cli.py`:

- top-level CLI surface:
  `biucing list`, `biucing info <template>`, `biucing create <template> <project-name>`, and `biucing --version`
- available templates:
  `frontend`, `web-service`, `microservice`, `apple`, `android`
- template-system behavior:
  metadata-driven template loading, required-variable prompting, `default` and `default_from` resolution, placeholder rendering, and executable-bit preservation for scripts and `gradlew`
- template-specific derived behavior:
  Apple platform presets for `ios`, `macos`, `watchos`, and `tvos`; microservice dependency presets for `postgres` and `redis`
- quality baseline:
  16 Python unit tests currently pass, covering the visible CLI surface plus rendered output for all five templates

## Capability Inventory

### CLI Product Surface

- `list` prints a summary of all available templates.
- `info` prints stack, variables, and next steps for a selected template.
- `create` generates a project into `--output-dir` and supports template-specific flags.
- `--version` reports the installed CLI version.

### Template Portfolio

- `frontend`:
  React + TypeScript starter with Docker-based local development, Vitest, and Playwright smoke checks
- `web-service`:
  Go + Gin service starter with Docker dev/runtime workflows and lint/test/doctor commands
- `microservice`:
  Go + Gin + Protobuf + Buf + OpenTelemetry starter with gRPC and selectable `postgres` or `redis` local dependency wiring
- `apple`:
  SwiftUI + Tuist + SwiftPM starter supporting `ios`, `macos`, `watchos`, and `tvos`, with platform-aware generated output
- `android`:
  Kotlin + Gradle + Jetpack Compose starter with committed Gradle wrapper, modularized structure, fastlane, and UI smoke coverage

### Where The Product Is Strong

- the template portfolio is already differentiated and practical;
- the metadata model is intentionally simple and readable;
- generated projects expose meaningful developer workflows instead of placeholder files only;
- the repo already has real validation evidence behind the major template families.

### Where The Product Is Thin

- the CLI has no machine-readable output for scripting or automation;
- template maturity and validation status are described in docs, not exposed as first-class metadata;
- there is no repo-level command for validating template definitions or generated output expectations;
- release planning artifacts have lagged behind implementation progress;
- the current roadmap was phase-based and no longer matched shipped reality.

## 0.3.0 Theme

`0.3.0` should focus on productizing BiucingCLI itself.

The main idea is:

- keep the five current templates stable;
- avoid chasing more breadth for one release;
- strengthen discoverability, automation, verification, and release discipline around the generator.

## Proposed Scope

### 1. CLI Experience Hardening

Goal:
make BiucingCLI easier to inspect and safer to automate.

Candidate deliverables:

- add `--json` output to `list` and `info`
- add a non-interactive creation path that fails clearly when required values are missing
- add a uniform `--set key=value` style override path so template creation is easier to script
- improve create-time summaries so users can see resolved values and important defaults

Why it matters:
the current CLI is friendly for manual use, but still thin for repeatable automation and tool-to-tool integration.

### 2. Metadata Evolution

Goal:
turn more of the template story into structured metadata instead of prose only.

Candidate deliverables:

- extend `template.json` with fields such as `category`, `platforms`, `maturity`, `validation`, and `tags`
- expose those fields in `list` and `info`
- distinguish required user inputs from derived values and recommended defaults more clearly

Why it matters:
the templates already have meaningful differences, but the CLI cannot explain them cleanly yet.

### 3. Verification Productization

Goal:
make template quality easier to prove and maintain release over release.

Candidate deliverables:

- add repo-level validation for template metadata completeness and placeholder consistency
- add snapshot or golden checks for `list` and `info` output where useful
- document a verification matrix for each template's minimum release bar
- add a repeatable release checklist for template rendering tests plus generated-project smoke evidence

Why it matters:
`0.2.0` proved the templates with real runs, but the proof is still scattered across tests, docs, and prior release work.

### 4. Release Surface Cleanup

Goal:
make future version planning and release communication easier to maintain.

Candidate deliverables:

- keep roadmap and version plans versioned instead of phase-only
- prepare a reusable `0.x` release checklist artifact
- define what belongs in `CHANGELOG.md` versus deeper planning docs

Why it matters:
the codebase moved faster than the planning documents; `0.3.0` is a good point to fix that operational gap.

## Non-Goals

These should stay out of the core `0.3.0` scope unless priorities change:

- adding many new template families
- replacing the simple renderer with a heavy template engine
- introducing remote registries, plugin systems, or agent workflow orchestration

## Recommended Backlog Order

1. Add structured metadata extensions and keep all five templates aligned.
2. Extend CLI output and creation flags to use the richer metadata.
3. Add repo-level validation for metadata and placeholder consistency.
4. Add release-checklist and verification-matrix docs.
5. Cut the `0.3.0` release only after the updated CLI flows and checks are covered by tests.

## Exit Criteria

`0.3.0` should be considered ready when:

- the five existing templates still generate successfully through tests;
- the CLI can expose template details in both human-readable and machine-readable forms;
- metadata is rich enough to communicate maturity and validation expectations;
- the repo has a clear, current release plan and verification story for the version.

## Suggested Changelog Direction

If this scope lands, the `0.3.0` changelog will likely read as a product-hardening release:

- improved BiucingCLI inspection and scripting workflows
- richer template metadata and clearer maturity signals
- stronger repo-level validation and release planning discipline
