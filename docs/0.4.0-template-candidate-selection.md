# BiucingCLI 0.4.0 Template Candidate Selection

## Purpose

This document closes the `Phase A` template-selection task from the `0.4.0` plan.

The repo should enter implementation with one chosen direction for `New Template Surface`, not an open-ended set of possibilities.

## Selection Criteria

The `0.4.0` plan said the chosen template should:

- fit the maintainer's real working stack;
- reuse patterns already proven in the repo;
- support believable generated-project validation;
- add real portfolio coverage instead of overlapping heavily with an existing template;
- avoid requiring a heavy new template-engine feature just to exist.

## Candidates Reviewed

| Candidate | Summary | Strengths | Main concerns |
| --- | --- | --- | --- |
| `worker` | background job or task-runner starter | complements `web-service` and `microservice`; reuses Go/Docker/Makefile patterns; useful for scheduled jobs, consumers, and async work | needs a clear boundary so it does not turn into a second microservice |
| `cli` | command-line application starter | aligns with repo productivity bias; could be attractive for internal tools | weaker alignment with the current starter portfolio and current language mix |
| `deployment` or infra helper | narrow deployment-oriented starter | could support the web backends operationally | high risk of becoming configuration-heavy and less reusable as a general project starter |

## Chosen Direction

`0.4.0` should add a `worker` template.

## Why `worker` Wins

### 1. It complements the current portfolio cleanly

The current product has:

- `frontend` for browser delivery,
- `web-service` for request/response APIs,
- `microservice` for contract-first service boundaries,
- `apple` and `android` for native clients.

What is missing is a first-class starter for asynchronous or background execution:

- scheduled jobs,
- queue consumers,
- maintenance tasks,
- long-running non-HTTP process workloads.

That gap is real without being artificial.

### 2. It reuses proven repo patterns

A `worker` starter can borrow from the existing backend family:

- Go as the base language;
- `Makefile`-driven workflows;
- Docker dev/runtime packaging;
- repo-local cache patterns where appropriate;
- config file plus environment override structure;
- health or readiness semantics where useful.

This means the new surface can grow from existing product habits rather than inventing a new worldview.

### 3. It adds new coverage without forcing a new engine model

A `worker` starter does not require:

- conditional template inheritance,
- remote template resolution,
- profile explosion,
- or a large new metadata abstraction.

It can fit the current metadata-driven rendering model comfortably.

## What `worker` Should Mean

To avoid overlap with `microservice`, the template should stay narrow.

The default `worker` concept should be:

- a background execution service;
- no public HTTP API as the central purpose;
- one or two execution modes such as queue consumer or scheduled loop;
- operational basics such as config, logging, graceful shutdown, and container packaging.

It should not try to become:

- another gRPC service template;
- a workflow orchestration platform;
- a message-bus abstraction showcase.

## Initial Design Boundaries

The `worker` template should likely include:

- Go-based project structure;
- `Makefile`, `README.md`, `.gitignore`, `.dockerignore`;
- Docker dev/runtime flow;
- config file plus environment override path;
- a main process entrypoint under `cmd/worker`;
- internal packages for config, runtime, and task execution;
- a verification path grounded in tests plus at least one believable local run.

The `worker` template should probably avoid in `0.4.0`:

- supporting several queue backends at once;
- carrying both scheduler and queue-consumer specializations as first-class modes;
- embedding OpenTelemetry unless it is truly lightweight and justified;
- pretending to support every async workload pattern.

## Deferred Candidates

### `cli`

Why it is deferred:

- attractive, but it pulls the portfolio toward internal-tool scaffolds rather than application/runtime starters;
- less obviously connected to the current product story than `worker`.

### deployment or infra helper

Why it is deferred:

- more likely to become environment-specific and configuration-heavy;
- weaker fit for the current "project starter" promise.

## Follow-On Tasks For Phase D

Before implementation starts, `worker` still needs:

1. a concrete variable set;
2. a generated directory shape;
3. a validation story that is comparable to the existing backend starters;
4. a decision on whether the first version is scheduler-oriented, consumer-oriented, or a narrow hybrid.

## Phase A Decision Summary

The new template-surface decision for `0.4.0` is:

- ship one new template only;
- that template should be `worker`;
- keep it narrow, backend-adjacent, and reusable;
- defer `cli` and deployment-focused starters to later releases.
