# BiucingCLI 0.3.0 Release Prep

This is the concrete execution checklist for preparing `0.3.0`.

It is intentionally narrower than [release-checklist.md](release-checklist.md):

- this file is version-specific;
- the checklist uses the current repo state and the known `0.3.0` scope;
- once `0.3.0` ships, this file can become historical evidence or be replaced by the next version-prep file.

Use this together with:

- [docs/0.3.0-plan.md](0.3.0-plan.md)
- [docs/release-checklist.md](release-checklist.md)
- [docs/verification-matrix.md](verification-matrix.md)

## 1. Confirm The 0.3.0 Story

Before editing version numbers, confirm the release narrative still matches the code:

- template metadata was expanded and is now exposed through the CLI;
- `biucing list --json` and `biucing info --json` are shipped;
- `create` now supports `--set key=value` and `--non-interactive`;
- repo-level validation exists for metadata and placeholder consistency;
- `list/info` golden checks are part of the automated test surface;
- release docs and verification docs are now present in the repo.

If any item above is not actually true in the checked-out tree, fix the product or trim the release claim before bumping the version.

## 2. Version Touchpoints For 0.3.0

Update these files in one pass:

| File | 0.3.0 action |
| --- | --- |
| `README.md` | Change the release target from `0.2.0` to `0.3.0` and keep the feature summary current |
| `CHANGELOG.md` | Add `0.3.0 - YYYY-MM-DD` above `0.2.0` |
| `pyproject.toml` | Set `[project].version = "0.3.0"` |
| `src/biucingcli/__init__.py` | Set `__version__ = "0.3.0"` |
| `tests/test_cli.py` | Update the expected `biucing 0.3.0` version string |
| `docs/0.3.0-plan.md` | Review whether any planned items now need to be marked as completed or tightened |

## 3. Recommended 0.3.0 Command Sequence

Run these in order once the version surfaces are updated:

```bash
python3 -m unittest discover -s tests
PYTHONPATH=src python3 -m biucingcli.cli validate
PYTHONPATH=src python3 -m biucingcli.cli list
PYTHONPATH=src python3 -m biucingcli.cli list --json
PYTHONPATH=src python3 -m biucingcli.cli info web-service
PYTHONPATH=src python3 -m biucingcli.cli info web-service --json
```

Then do at least one scriptability smoke:

```bash
PYTHONPATH=src python3 -m biucingcli.cli create web-service demo-service --output-dir /tmp/biucing-0.3.0-check --non-interactive --set project_name=demo-service --set module_name=github.com/example/demo-service
```

If the release includes changes to shared create flow, metadata plumbing, or placeholder resolution, also run one native-template smoke:

```bash
PYTHONPATH=src python3 -m biucingcli.cli create apple demo-apple --output-dir /tmp/biucing-0.3.0-check --non-interactive --set project_name=demo-apple --set bundle_identifier=com.example.demoapple
```

## 4. Minimum 0.3.0 Fresh Proof

Because `0.3.0` is about productizing the generator rather than adding new template families, the minimum practical fresh-proof bar should be:

- repo-wide automated checks all pass;
- one Dockerized template is regenerated through the new scriptable path;
- one native template is regenerated through the new scriptable path;
- `validate` stays clean;
- `list/info` output remains stable under golden checks.

Recommended pair:

- Dockerized template: `web-service`
- Native template: `apple`

If `android`-specific or `microservice`-specific flows changed materially, substitute the touched template instead of following the default pair mechanically.

## 5. Draft Changelog Shape For 0.3.0

The changelog should stay user-facing. A good `0.3.0` entry will likely emphasize:

- richer template metadata and clearer maturity/validation signals;
- machine-readable `list/info` output for scripting;
- scriptable `create` improvements through `--set` and non-interactive failure behavior;
- stronger repo-level validation and output-golden protection;
- better release and verification documentation.

Avoid overfilling the changelog with internal refactor details unless they changed the user-facing CLI or release bar.

## 7. Final Release Command Block

Once the repo is verified and the changelog is final:

```bash
git status --short
git add README.md CHANGELOG.md pyproject.toml src/biucingcli/__init__.py tests/test_cli.py docs/0.3.0-plan.md docs/release-checklist.md docs/verification-matrix.md docs/0.3.0-release-prep.md
git commit -m "Prepare release 0.3.0"
git tag -a v0.3.0 -m "Release 0.3.0"
git push origin main --follow-tags
gh release create v0.3.0 --title "BiucingCLI 0.3.0" --notes-file CHANGELOG.md
```

Use a narrower `git add` if the working tree contains unrelated local experimentation.
