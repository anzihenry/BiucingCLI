# BiucingCLI 0.4.0 Template Consistency Audit

## Purpose

This audit closes the `Phase A` consistency-review task from the `0.4.0` plan.

The goal is not to force every starter into the same shape.
The goal is to define which differences are intentional product choices and which differences should become consistency work in `0.4.0`.

## Audit Inputs

Reviewed sources:

- `templates/*/template.json`
- top-level generated file layout under each `templates/*/template/`
- current README and roadmap framing

Templates reviewed:

- `frontend`
- `web-service`
- `microservice`
- `apple`
- `android`

## Shared Contract Already Present

All five templates already share these product-level traits:

| Area | Current shared baseline | Notes |
| --- | --- | --- |
| Metadata file | every template ships `template.json` | this is the source of truth for CLI discovery and validation |
| Core metadata | every template has `name`, `description`, `category`, `stack`, `tags`, `platforms`, `maturity`, `validation`, `variables`, `next_steps` | this is already strong enough to treat metadata as a real contract |
| Readme surface | every template ships `README.md` | docs depth varies, but the existence of starter-facing docs is consistent |
| Entry workflow | every template exposes `Makefile` and actionable next steps | naming differs, but each starter tries to give a real first-run path |
| Ignore/config hygiene | every template ships `.gitignore` | file families differ by ecosystem, which is expected |
| Validation signal | every template declares maturity and validation evidence | evidence wording is not normalized yet |

## Top-Level Shape Snapshot

| Template | Top-level workflow files | Distinctive traits |
| --- | --- | --- |
| `frontend` | `Dockerfile`, `Dockerfile.dev`, `Dockerfile.dev.full`, `compose.dev.yaml`, `Makefile` | strongest browser-facing dev/runtime split |
| `web-service` | `Dockerfile`, `Dockerfile.dev`, `compose.dev.yaml`, `compose.yaml`, `.air.toml`, `Makefile` | Go service baseline with live reload and runtime image variants |
| `microservice` | `Dockerfile`, `Dockerfile.dev`, `compose.dev.yaml`, `.air.toml`, `Makefile`, `deploy/` | contract-first backend with Buf, gRPC, OTel, dependency store |
| `apple` | `Tuist.swift`, `Workspace.swift`, `Makefile`, `scripts/` | native multi-platform starter with Tuist + SwiftPM structure |
| `android` | `gradlew`, `gradlew.bat`, `Makefile`, `scripts/`, `fastlane/` | native mobile starter with committed wrapper and richer release surface |

## Intentional Differences

These differences should be preserved, not flattened away:

| Difference | Why it is intentional |
| --- | --- |
| Web templates are Docker-first while Apple/Android are not | this matches actual platform constraints rather than an inconsistency bug |
| `apple` and `android` carry stronger platform bootstrap/setup files | native environments genuinely need more workstation guidance |
| `microservice` has `api/`, `deploy/`, and observability wiring | it is deliberately broader than `web-service` because it targets a different backend shape |
| `frontend` has both light and full dev Dockerfiles | browser tooling makes the split useful and grounded in current workflows |

## Accidental or Under-Specified Differences

These are the main gaps `0.4.0` should address under `Template Consistency`:

| Area | Current situation | Why it matters in 0.4.0 |
| --- | --- | --- |
| Validation terminology | `generated-project-verified` and `real-build-verified` both exist, but without a documented tier model | users and maintainers need clearer semantics for what each level proves |
| Next-step naming | some templates center `make doctor`, some emphasize `make dev`, some include GUI open commands | the workflow is useful, but the product vocabulary is not fully normalized |
| Bootstrap contract | some templates clearly expose `scripts/bootstrap` and `scripts/doctor`; others rely more on README explanation | validate can enforce more of the intended starter contract |
| Runtime/deploy wording | web templates use overlapping terms like `docker-build`, `docker-run`, `up`, and `verify` with slightly different meanings | the current commands work, but cross-template expectations are harder to learn |
| Metadata extensibility | metadata is already rich, but it does not yet describe operating assumptions or verification tier explicitly | the repo is ready for one more layer of structured consistency rules |

## Proposed 0.4.0 Minimum Product Contract

The following should become the minimum shared contract for starters in this repo:

### Required metadata

Every template should continue to require:

- `name`
- `description`
- `category`
- `stack`
- `tags`
- `platforms`
- `maturity`
- `validation`
- `variables`
- `next_steps`

### Candidate metadata additions

`0.4.0` should evaluate adding:

- `operating_assumptions`
- `verification_tier`
- `workflow_labels` or a similarly small mechanism for normalizing starter workflows

### Required starter-facing files by family

These should not become one-size-fits-all global rules, but they are strong candidates for validation policy:

| Template family | Candidate required files/directories |
| --- | --- |
| web/container starters | `README.md`, `Makefile`, `.gitignore`, `.dockerignore`, dev Dockerfile, runtime Dockerfile or explicit runtime packaging path |
| Go service starters | `README.md`, `Makefile`, `go.mod`, `go.sum`, `cmd/`, `internal/`, `configs/`, `scripts/` |
| native starters | `README.md`, `Makefile`, `.mise.toml`, `scripts/`, platform build entrypoints |

### Required workflow labels

`0.4.0` should try to normalize these user-facing workflow concepts where they exist:

- bootstrap
- doctor
- dev
- test
- verify
- build
- release or package

The command implementations can differ by ecosystem, but the vocabulary should become more predictable.

## Workstream Recommendations

### Highest-value consistency tasks

1. Document and enforce a validation-tier model.
2. Normalize next-step and workflow vocabulary across templates.
3. Expand `validate` to check family-level required files.
4. Update template-system docs so new templates inherit the same contract.

### Tasks to avoid overdoing

- forcing Apple/Android into Docker-like conventions;
- forcing all templates to expose exactly the same Make targets;
- creating a large standalone schema system separate from the current renderer contract.

## Phase A Decision Summary

`Phase A` concludes that the template family is already structurally strong.
The best `0.4.0` consistency work is not a rewrite.
It is a contract-tightening pass that:

- clarifies validation tiers;
- standardizes workflow vocabulary;
- promotes family-level file expectations into validation rules;
- makes future template additions conform to the same bar from day one.
