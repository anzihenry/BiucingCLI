# BiucingCLI 0.4.0 Release Prep

This is the concrete execution checklist for preparing `0.4.0`.

It is intentionally narrower than [release-checklist.md](release-checklist.md):

- this file is version-specific;
- the checklist uses the current repo state and the known `0.4.0` scope;
- once `0.4.0` ships, this file can become historical evidence or be replaced by the next version-prep file.

Use this together with:

- [docs/0.4.0-plan.md](0.4.0-plan.md)
- [docs/release-checklist.md](release-checklist.md)
- [docs/verification-matrix.md](verification-matrix.md)

## 1. Confirm The 0.4.0 Story

Before editing version numbers, confirm the release narrative still matches the code:

- `create` now supports `--dry-run`, `--plan`, and JSON preview / manifest output;
- non-interactive failures now report all missing required values together;
- template metadata now exposes verification tier, operating assumptions, and workflow labels;
- repo-level validation now checks the stronger template contract plus family-level required starter entries;
- the new `worker` template is shipped as a sixth starter;
- generated-project validation now includes real `go test ./...` proof for the `worker` starter.

If any item above is not actually true in the checked-out tree, fix the product or trim the release claim before bumping the version.

## 2. Version Touchpoints For 0.4.0

Update these files in one pass:

| File | 0.4.0 action |
| --- | --- |
| `README.md` | Change the release target from `0.3.0` to `0.4.0` and keep the feature summary current |
| `CHANGELOG.md` | Replace the draft `Unreleased` notes with a dated `0.4.0 - YYYY-MM-DD` entry |
| `pyproject.toml` | Set `[project].version = "0.4.0"` |
| `src/biucingcli/__init__.py` | Set `__version__ = "0.4.0"` |

| `docs/0.4.0-plan.md` | Review whether any phase-status notes or claims need tightening before release |

## 3. Recommended 0.4.0 Command Sequence

Run these in order once the version surfaces are updated:

```bash
python3 -m unittest discover -s tests
PYTHONPATH=src python3 -m biucingcli.cli validate
PYTHONPATH=src python3 -m biucingcli.cli list
PYTHONPATH=src python3 -m biucingcli.cli list --json
PYTHONPATH=src python3 -m biucingcli.cli info web-service
PYTHONPATH=src python3 -m biucingcli.cli info web-service --json
PYTHONPATH=src python3 -m biucingcli.cli info worker
PYTHONPATH=src python3 -m biucingcli.cli info worker --json
```

Then do the minimum Generator UX smoke checks:

```bash
PYTHONPATH=src python3 -m biucingcli.cli create frontend preview-app --output-dir /tmp/biucing-0.4.0-check --dry-run
PYTHONPATH=src python3 -m biucingcli.cli create web-service plan-service --output-dir /tmp/biucing-0.4.0-check --module-name github.com/example/plan-service --plan --json
```

Then do at least one real scriptable create per changed starter family:

```bash
PYTHONPATH=src python3 -m biucingcli.cli create web-service demo-service --output-dir /tmp/biucing-0.4.0-check --non-interactive --set project_name=demo-service --set module_name=github.com/example/demo-service
PYTHONPATH=src python3 -m biucingcli.cli create worker demo-worker --output-dir /tmp/biucing-0.4.0-check --non-interactive --set project_name=demo-worker --set module_name=github.com/example/demo-worker
```

## 4. Minimum 0.4.0 Fresh Proof

Because `0.4.0` changes both the generator surface and the shipped starter portfolio, the minimum practical fresh-proof bar should be:

- repo-wide automated checks all pass;
- at least one Dockerized starter is regenerated through the new preview / create path;
- the new `worker` starter is freshly generated and re-tested in its generated project directory;
- `validate` stays clean under the stronger template contract;
- `info` and JSON inspection output still reflect the intended product surface.

Recommended pair:

- Dockerized existing starter: `web-service`
- New starter: `worker`

If the final release touches native-template output again, add one native-template smoke instead of assuming existing evidence is enough.

## 5. Draft Changelog Shape For 0.4.0

The changelog should stay user-facing. A good `0.4.0` entry will likely emphasize:

- generator UX improvements through `--dry-run`, `--plan`, and JSON create manifests;
- stronger template metadata and repo-level contract validation;
- the new `worker` starter for background execution workloads;
- updated release and verification documentation for the expanded product surface.

Avoid overfilling the changelog with internal implementation details unless they changed the user-facing CLI or release bar.

## 7. Final Release Command Block

Once the repo is verified and the changelog is final:

```bash
git status --short
git add README.md CHANGELOG.md pyproject.toml src/biucingcli/__init__.py tests/test_cli.py docs/0.4.0-plan.md docs/release-checklist.md docs/verification-matrix.md docs/0.4.0-release-prep.md
git commit -m "Prepare release 0.4.0"
git tag -a v0.4.0 -m "Release 0.4.0"
git push origin main --follow-tags
gh release create v0.4.0 --title "BiucingCLI 0.4.0" --notes-file CHANGELOG.md
```

Use a narrower `git add` if the working tree contains unrelated local experimentation.
