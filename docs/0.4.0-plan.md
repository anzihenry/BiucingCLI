# BiucingCLI 0.4.0 Plan

## Current State

BiucingCLI `0.3.0` has moved the project beyond a template collection and into a usable generator product:

- the CLI now exposes `list`, `info`, `create`, `validate`, and `--version`;
- `list`, `info`, and `validate` support both human-readable and JSON output;
- `create` supports interactive prompting, `--set KEY=VALUE`, and `--non-interactive`;
- template metadata now includes category, stack, tags, platforms, maturity, validation, variables, and next steps;
- repo-level validation checks metadata completeness and placeholder consistency from the same source of truth used by rendering;
- the current five-template portfolio remains stable:
  `frontend`, `web-service`, `microservice`, `apple`, and `android`.

This is a stronger base than `0.2.0`, but it also makes the next gap more obvious:
the generator surface is useful, yet still light in planning, preview, and lifecycle support; the templates are stronger, yet not fully normalized as one product family; and the template portfolio is focused, yet still has clear room for one carefully chosen expansion.

## 0.4.0 Theme

`0.4.0` should be the release where BiucingCLI grows from a solid scaffold generator into a more inspectable, more consistent, and more intentionally extensible product.

The release should use three equal planning axes:

- `Generator UX`
- `Template Consistency`
- `New Template Surface`

The goal is not to maximize raw scope.
The goal is to make the current product easier to trust, easier to extend, and easier to operate release after release.

Completed artifacts:

## Phase B Status

`Phase B` is complete in implementation terms.

Shipped generator UX improvements:

- `biucing create ... --dry-run` now previews generation without writing files;
- `biucing create ... --plan` can emit a structured preview, including JSON output with `--json`;
- successful `create --json` now returns a machine-readable generation manifest;
- non-interactive failures now report all missing required values in one pass instead of stopping at the first missing field.

## Phase C Status

`Phase C` is complete in implementation terms.

Shipped template-consistency improvements:

- template metadata now includes `validation.verification_tier`, `operating_assumptions`, and `workflow_labels`;
- `info` and JSON metadata surfaces now expose those fields directly;
- repo-level validation now checks the stronger metadata contract plus family-level required starter entries;
- template-system docs now describe the current contract and validation policy instead of the older minimal schema.

## Planning Principles

- keep the current five templates healthy while adding new capability;
- prefer features that improve repeated use, not one-off demo value;
- preserve the readable internal template system instead of introducing a heavy abstraction layer;
- treat new template expansion as a product decision, not a template-count exercise;
- require every scope item to improve either discoverability, maintainability, or generated-project usefulness.

## Scope Overview

| Theme | Release Intent | Success Signal |
| --- | --- | --- |
| Generator UX | make generation easier to inspect, preview, and automate | users can understand what will be generated before files land, and can audit what was generated after |
| Template Consistency | make the template family behave more like one product with shared quality expectations | template metadata, validation, docs, and developer workflows feel intentionally aligned |
| New Template Surface | add one carefully selected new starter or starter family without diluting the product | the new surface fits the current stack philosophy and reaches the same quality bar as existing templates |

## Workstream 1: Generator UX

### Problem

BiucingCLI can already generate projects reliably, but the current flow is still minimal:

- there is no preview mode for resolved values and planned file output;
- there is no machine-readable generation result manifest;
- there is no easy way to inspect required inputs before running a real create flow;
- the current UX is strong enough for maintainers, but still thin for repeatable automation and tooling integration.

### 0.4.0 Goal

Make generation inspectable before execution and auditable after execution.

### Candidate Deliverables

#### 1. Create Preview Layer

- add `biucing create ... --dry-run` to resolve variables without writing files;
- add `biucing create ... --plan` or an equivalent preview mode that shows:
  - selected template;
  - resolved variables;
  - derived variables;
  - target directory;
  - key generated workflows or next steps;
- make the preview available in both human-readable and JSON forms.

#### 2. Input Discovery Improvements

- add a way to inspect a template's required and optional inputs more directly from the CLI;
- improve missing-value errors so non-interactive failures list all missing required values together;
- consider a focused `info` enhancement rather than introducing too many new top-level commands.

#### 3. Generation Result Manifest

- emit a machine-readable summary after create succeeds;
- include at minimum:
  - template name;
  - project name;
  - output path;
  - resolved variables;
  - key next steps;
  - possibly file count or important generated entrypoints;
- decide whether this is stdout JSON, a saved manifest file, or both.

### Task Breakdown

#### Phase 1. Preview and Error Design

- define the output shape for dry-run and plan modes;
- define how resolved values should distinguish user-provided, derived, and defaulted fields;
- decide whether preview belongs under `create` flags or a separate inspection command;
- write tests for failure messaging and preview payload shape first.

#### Phase 2. CLI Implementation

- extend parser support for preview-oriented flags;
- reuse current resolution logic so preview and real generation cannot drift;
- implement human-readable preview formatting;
- implement JSON preview formatting.

#### Phase 3. Result Manifest

- define a stable manifest structure for successful generation;
- add tests for manifest shape and field semantics;
- document how scripts should rely on it.

## Workstream 2: Template Consistency

### Problem

The current templates are individually strong, but the product still shows mild unevenness:

- validation evidence is structured, but not yet normalized into stronger release-level expectations;
- next steps are useful, but not clearly standardized across template families;
- template docs and generated workflows differ in depth and naming conventions;
- some quality expectations still live in repository knowledge rather than in explicit product rules.

### 0.4.0 Goal

Make the template family feel intentionally governed rather than historically accumulated.

### Candidate Deliverables

#### 1. Metadata Contract Tightening

- define which metadata fields are mandatory for every template release;
- consider additional metadata such as:
  - operating assumptions;
  - external dependency requirements;
  - verification tier;
  - generated-project maturity notes;
- document the contract clearly so new templates follow the same bar.

#### 2. Validation Policy Expansion

- extend `validate` beyond metadata completeness and placeholder consistency where it adds real value;
- consider checks for:
  - required top-level files by template family;
  - expected docs such as `README.md`, `Makefile`, and doctor/bootstrap scripts where relevant;
  - consistency between template metadata and generated workflows;
  - validation evidence shape and terminology.

#### 3. Shared Output Standards

- normalize how templates describe:
  - bootstrap;
  - doctor;
  - test;
  - verification;
  - release or runtime workflows;
- align help text, `info` output, README expectations, and generated Make targets more deliberately.

### Task Breakdown

#### Phase 1. Consistency Audit

- compare all five templates against one shared checklist;
- identify differences that are intentional versus accidental;
- write down the minimum common product contract for starters in this repo.

#### Phase 2. Validation and Metadata Enhancements

- update `template.json` expectations where needed;
- extend validation logic carefully without creating a second schema system;
- add tests that lock the new rules in place.

#### Phase 3. Template Surface Alignment

- normalize template docs and next steps;
- tighten naming consistency across doctor/bootstrap/verify flows;
- refresh README or template-system docs to describe the stronger contract.

## Workstream 3: New Template Surface

### Problem

BiucingCLI now has a focused portfolio, which is a strength.
At the same time, if `0.4.0` only hardens the existing product surface, the release may undersell the generator's ability to grow thoughtfully.

### 0.4.0 Goal

Ship one new template surface that feels inevitable for this product, not opportunistic.

### Scope Rule

`0.4.0` should add at most one new starter line or one tightly related starter family.
Anything broader risks turning the release into parallel template bring-up instead of product progress.

### Candidate Evaluation Criteria

Any new template candidate should satisfy most of these:

- fits the maintainer's real working stack;
- reuses patterns already proven in the repo;
- can be validated with a believable generated-project workflow;
- adds genuine portfolio coverage rather than overlapping heavily with an existing template;
- does not require a heavy new template engine feature just to exist.

### Candidate Directions To Evaluate

- a backend-adjacent template that complements `web-service` and `microservice` without duplicating them;
- a supporting service or worker template;
- a CLI or automation-oriented starter that matches the repo's productivity bias;
- a deployment-facing or infrastructure-adjacent starter only if it remains narrow and practical.

### Task Breakdown

#### Phase 1. Candidate Selection

- define 2-3 realistic template candidates;
- compare them against the evaluation criteria;
- choose one winner explicitly and record why the others are deferred.

Status:
complete for planning.
See [0.4.0 Template Candidate Selection](0.4.0-template-candidate-selection.md).

#### Phase 2. Design Spec

- define template metadata, variables, generated structure, and next steps;
- define what "validated" means for the new template before implementation starts;
- decide whether any new placeholders or derived-value logic are truly required.

#### Phase 3. Implementation and Validation

- add CLI wiring only if the existing metadata-driven system needs it;
- add rendering tests and generated-project verification steps;
- update docs, roadmap surfaces, and template inventory in one pass.

## Non-Goals

The following should stay out of the core `0.4.0` scope unless priorities change:

- adding several unrelated new templates in one release;
- replacing the simple renderer with a full templating engine;
- adding remote registries, online template fetching, or plugin ecosystems;
- turning BiucingCLI into a workflow orchestrator rather than a scaffold generator;
- deep upgrade/migration support for previously generated projects beyond lightweight manifests or guidance.

## Exit Criteria

`0.4.0` should be considered ready when:

- BiucingCLI can preview or dry-run generation in a trustworthy way;
- create-time automation is more inspectable before generation and more auditable after generation;
- template metadata and validation reflect a clearer shared product contract;
- one new template surface is added without lowering the repo's quality bar;
- roadmap, docs, and release surfaces match shipped reality again.

## Suggested Changelog Direction

If this scope lands, the `0.4.0` changelog should read as a product-expansion release with discipline:

- stronger generation preview and automation UX;
- tighter template quality and consistency rules;
- one thoughtfully chosen new template surface added to the portfolio.
