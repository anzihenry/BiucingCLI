## Repository Layout

- `app/`: installable Android application module
- `feature/`: user-facing feature modules
- `core/`: shared modules such as models and design system
- `gradle/`: version catalog and wrapper configuration
- `fastlane/`: beta and release automation
- `scripts/`: bootstrap and environment checks

## Getting Started

```bash
make bootstrap
make build
make test
```

If the team later upgrades Gradle and needs to refresh the wrapper files, run:

```bash
make wrapper
```

Then commit the updated `gradle/wrapper/gradle-wrapper.jar`, `gradle-wrapper.properties`, and launcher scripts.

## Standard Commands

```bash
make doctor
make build
make test
make lint
make install-debug
make beta
```

## Notes

- `settings.gradle.kts` defines the module graph.
- `gradle/libs.versions.toml` is the source of truth for plugin and dependency versions.
- `app/` should stay focused on app shell concerns.
- `feature/home` demonstrates how to add a user-facing feature module.
- `gradle/wrapper/gradle-wrapper.jar` is committed in the starter and should stay versioned in the repo.
- Release signing is intentionally not configured in the local starter by default.
